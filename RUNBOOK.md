## Daily Operations

### Trigger a manual daily screen

```bash
# CLI
python scripts/run_daily.py --date today

# Make target (equivalent)
make daily

# Specific past date
python scripts/run_daily.py --date 2026-04-10

# Watchlist only (skip full universe)
python scripts/run_daily.py --date today --watchlist-only

# Inline symbols only
python scripts/run_daily.py --date today --symbols "DIXON,TATAELXSI,CDSL"
```

### Check if last night's run succeeded

**Query `run_history` directly:**
```bash
sqlite3 data/minervini.db \
  "SELECT run_id, run_date, status, duration_sec, a_plus_count, a_count, error_msg
   FROM run_history ORDER BY run_id DESC LIMIT 5;"
```

- `status = "success"` → clean run
- `status = "partial"` → `run_screen` failed; results were empty; check `error_msg`
- `status = "failed"` → pipeline aborted early; check logs

**Check systemd journal (if running via systemd timer):**
```bash
journalctl -u minervini-daily.service --since "yesterday" --no-pager
# Or tail the rotating log file:
tail -100 logs/minervini.log | grep -E "ERROR|WARNING|Pipeline run complete"
```

### Re-run a failed screen for a specific past date

```bash
# Re-run features + screen for 2026-04-09 (safe — update() is idempotent
# for dates not yet in the feature store; raises FeatureStoreOutOfSyncError
# for dates already present, which runner.py logs and skips gracefully)
python scripts/run_daily.py --date 2026-04-09

# If the feature store already has that date and you want to force a fresh screen
# without recomputing features (features are already up to date):
python scripts/run_daily.py --date 2026-04-09 --skip-features
```

> **Note:** The runner is self-healing. If a symbol's feature file is missing
> for the target date, it auto-bootstraps before screening.

## Paper Trading

Paper trading runs the full Minervini screening pipeline in **simulation mode** — positions are recorded in the local SQLite database, but no real orders are placed. Use it to validate signal quality against live market conditions before committing capital.

### Enable paper trading

```bash
make paper-start
```

This sets `paper_trading.enabled: true` in `config/settings.yaml`. The next `make daily` run will begin routing qualifying signals through the paper-trading engine.

> **Important:** Paper trading only enters positions for **A+ or A quality setups with a score ≥ 70**. Lower-quality signals (`B`, `C`, `FAIL`) are screened and reported but not traded.

### Check current portfolio status

```bash
make paper-status
```

Prints a summary of open positions, unrealised P&L, cash balance, and the running return vs the initial capital. If the portfolio has not been initialised yet, or no trades have been executed, it prints an informational message rather than raising an error.

### Reset the portfolio

```bash
make paper-reset
```

Wipes all paper-trading positions and P&L history from `data/minervini.db` and re-initialises the portfolio with `initial_capital` as defined in `config/settings.yaml → paper_trading.initial_capital`.

Use this when starting a fresh evaluation window. **All historical paper-trade records are permanently deleted.**

### Market hours and signal timing

Signals are generated by the daily pipeline at **15:35 IST** (5 minutes after NSE close). Any buy signal produced at this time is **queued for the next market open at 09:15 IST** the following trading day. Entries are always simulated at the next-day open price.

### How long to run before drawing conclusions

> ⚠ **Run paper trading for at least 4–8 weeks before comparing its results to the backtester.**

Backtester results are computed on historical data with hundreds of setups across multiple market regimes. A short paper-trading window will have far fewer setups and will be heavily influenced by the market conditions of that specific period. Four to eight weeks gives enough setups (typically 10–30 trades depending on market conditions) for a statistically meaningful comparison.

### Configuration reference

Key settings in `config/settings.yaml`:

```yaml
paper_trading:
  enabled: false            # set to true via `make paper-start`
  initial_capital: 1000000  # ₹10 lakh default starting capital
  position_size_pct: 0.05   # 5 % of portfolio per position
  max_open_positions: 10    # cap on concurrent open trades
  min_score: 70             # only A+/A setups (score >= 70) are traded
```

## Adding a New Symbol to the Watchlist

### Via CLI (`--symbols` flag)

```bash
# Ad-hoc run only — symbol is NOT persisted to the watchlist table
python scripts/run_daily.py --date today --symbols "NEWSTOCK"

# To persist, insert directly into SQLite (see below)
```

### Via SQLite directly

```bash
sqlite3 data/minervini.db
```

```sql
INSERT INTO watchlist (symbol, note, added_via)
VALUES ('NEWSTOCK', 'Strong VCP forming', 'cli');
```

Full schema reference:
```sql
-- watchlist table columns
symbol       TEXT UNIQUE NOT NULL
note         TEXT                         -- optional
added_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP
added_via    TEXT  -- 'cli' | 'api' | 'dashboard' | 'file_upload'
last_score   REAL                         -- updated after each run
last_quality TEXT                         -- 'A+' | 'A' | 'B' | 'C' | 'FAIL'
last_run_at  TIMESTAMP
```

### Expected behaviour on next run

- Symbol is included in `run_symbols.watchlist` (resolved by `universe_loader.py`).
- Feature file is checked via `needs_bootstrap()`. If absent → auto-bootstraps (slow, ~5–15s per symbol).
- If feature file exists and is current → `update()` runs (fast, <1s).
- Result appears **first** in the HTML report with a "★ Watchlist" badge.
- Chart is always generated regardless of score (controlled by `watchlist.always_generate_charts: true` in `settings.yaml`).
- `update_symbol_score()` writes `last_score` and `last_quality` back to the `watchlist` table after the run.

## Recovering from a Corrupt Feature Store

### How to detect

- **Log symptom:** `WARNING Feature computation failed — skipping symbol` with `reason=` containing `ParquetWriteError`, `ArrowInvalid`, or `pyarrow` traceback.
- **Run symptom:** Symbol consistently absent from screen results despite being in universe.
- **Direct check:**
  ```bash
  python -c "
  import pandas as pd
  df = pd.read_parquet('data/features/DIXON.parquet')
  print(df.tail(3))
  print('Rows:', len(df))
  "
  # ArrowInvalid / EOFError / EmptyDataError → file is corrupt
  ```

### Rebuild one symbol

```bash
python scripts/rebuild_features.py --symbols DIXON
```

This runs `bootstrap(symbol, config)` → overwrites `data/features/DIXON.parquet` with a full recompute from `data/processed/DIXON.parquet`.

### Rebuild the full universe

```bash
# All symbols in config/universe.yaml
make rebuild

# Equivalent CLI
python scripts/rebuild_features.py --universe nifty500

# With progress output, estimate ~5–15 min for 500 symbols
```

> **Safe to run at any time.** `parquet_store.write(..., overwrite=True)` uses an atomic temp-file rename — the old file is never left in a partial state.

## Adding a New Rule Condition to the Trend Template

### 1. Add the condition in `rules/trend_template.py`

```python
# Inside check_trend_template(), add to the conditions dict:
conditions["new_condition_name"] = (
    row["some_feature"] >= config.new_threshold
)
```

- All 8 (now 9) conditions must pass for `trend_template_pass = True`.
- Use `row["feature_col"]` — the row is the last row of the feature DataFrame.
- `config` is a `TrendTemplateConfig` dataclass — add your new field there too.

### 2. Add the threshold in `config/settings.yaml`

```yaml
trend_template:
  ma200_slope_lookback: 20
  pct_above_52w_low: 25.0
  pct_below_52w_high: 25.0
  min_rs_rating: 70
  new_threshold: 0.5        # ← add here
```

### 3. Add the unit test in `tests/unit/test_trend_template.py`

- Follow the existing parametrized pattern: one test for pass, one for fail at the boundary.
- Add a row to the `@pytest.mark.parametrize` table if one exists, or add a new `test_new_condition_*` function.

### 4. Confirm no regression

```bash
# Run only trend template tests first
python -m pytest tests/unit/test_trend_template.py -v

# Then run the full integration regression suite
python -m pytest tests/integration/test_known_setups.py -v

# Full suite
make test
```

> **The 6 regression tests in `test_known_setups.py` are the authoritative gate.**  
> They cover: Stage 4 blocked despite all-8-TT-pass, A+ pipeline, partial-TT, etc.  
> All 6 must pass before merging.

## Adding a New Data Source

### 1. Implement `ingestion/base.py` `DataSource` interface

```python
# ingestion/my_new_source.py
from ingestion.base import DataSource
import pandas as pd

class MyNewSource(DataSource):
    def fetch(self, symbol: str, start, end) -> pd.DataFrame:
        # Return DatetimeIndex OHLCV DataFrame
        ...

    def fetch_universe(self) -> list[str]:
        # Return list of tradable symbols
        ...
```

Both methods must return data in the same schema as `yfinance_source.py`:  
columns `[open, high, low, close, volume]`, `DatetimeIndex`.

### 2. Register in `ingestion/universe_loader.py`

```python
# In the source factory / dispatch block:
from ingestion.my_new_source import MyNewSource

_SOURCES = {
    "yfinance":    YFinanceSource,
    "nse_bhav":    NSEBhavSource,
    "my_new_source": MyNewSource,   # ← add here
}
```

Then set in `config/settings.yaml`:
```yaml
universe:
  source: "my_new_source"
```

### 3. No other changes needed — here's why

- `ingestion/validator.py` validates any DataFrame that matches the OHLCV schema — source-agnostic.
- `features/feature_store.py` calls `parquet_store.read(processed_path)` — it never touches the source directly.
- `pipeline/runner.py` calls `resolve_symbols()` which calls the registered source — fully decoupled.
- The `DataSource` interface is the only contract the rest of the system depends on.

## Tuning Scoring Thresholds

### Keys in `settings.yaml` that control each weight

```yaml
scoring:
  weights:
    rs_rating:   0.30   # Relative Strength percentile rank (0–99)
    trend:       0.25   # Trend Template pass + condition density
    vcp:         0.25   # VCP grade (A/B/C) + contraction quality
    volume:      0.10   # Breakout volume ratio vs 50d avg
    fundamental: 0.07   # 7-condition fundamental template pass rate
    news:        0.03   # Rescaled news_score (−100..+100 → 0..1)

  min_score_alert: 70   # Minimum score to trigger Telegram/email alert

  setup_quality_thresholds:
    a_plus: 85    # score >= 85 → A+
    a:      70    # score >= 70 → A
    b:      55    # score >= 55 → B
    c:      40    # score >= 40 → C
                  # score < 40  → FAIL
```

All weights must sum to 1.0. Adjust `fundamental` and `news` weights only after enabling those modules.

### Run a backtest sweep to validate a threshold change

```bash
# Single backtest run
python scripts/backtest_runner.py --start 2021-01-01 --end 2024-01-01

# Make target
make backtest START=2021-01-01 END=2024-01-01

# Parameter sweep (e.g. testing different min_rs_rating values)
# Edit config, run backtest, compare HTML/CSV outputs in data/backtests/
# The backtest engine uses walk-forward to avoid lookahead bias.
python scripts/benchmark_features.py   # verify feature timing hasn't changed
```

Compare key metrics across runs: `CAGR`, `Sharpe`, `max_drawdown`, `win_rate` in the HTML report.

### ⚠ Warning: do not change thresholds mid-paper-trading window

- Paper trading results are only comparable within a single threshold regime.
- Changing `scoring.weights` or `setup_quality_thresholds` mid-window invalidates the P&L attribution.
- Freeze thresholds at the start of each paper-trading period. Record the config hash (`run_history.config_hash`) for that period before making changes.

## Common Errors & Fixes

### `InsufficientDataError`

- **Cause:** Feature computation (e.g. SMA_200) requires more rows than available in `data/processed/{symbol}.parquet`. Common for newly listed stocks or symbols with thin history.
- **Fix:** Check history length: `python -c "import pandas as pd; df=pd.read_parquet('data/processed/XYZ.parquet'); print(len(df))"`. If < 300 rows, the symbol cannot be fully featured yet — either exclude from universe via `min_listing_years` filter, or accept partial features. This error is caught and swallowed by `feature_store.bootstrap()` and `update()` — the symbol is skipped gracefully with a `WARNING` log.

### `RuleEngineError`

- **Cause:** A feature column expected by `rules/stage.py` or `rules/trend_template.py` is `NaN` or missing in the last row of the feature DataFrame (e.g. `SMA_200` is NaN because the processed file has exactly 200 rows with no buffer).
- **Fix:** Inspect the feature file: `python -c "import pandas as pd; print(pd.read_parquet('data/features/XYZ.parquet').tail(1).to_dict('records'))"`. If critical columns are `NaN`, rebuild the symbol: `python scripts/rebuild_features.py --symbols XYZ`. If the symbol consistently has too little history, add it to an exclusion list or increase `min_listing_years`.

### `FeatureStoreOutOfSyncError`

- **Cause:** `feature_store.update()` found that `run_date` is already present in the feature Parquet file. This is the idempotent guard — triggered when a daily run is executed twice for the same date.
- **Fix:** Nothing to do — this is expected behaviour. The runner logs a `WARNING` and skips the symbol. If you genuinely need to recompute today's features (e.g. after a data correction): `python scripts/rebuild_features.py --symbols XYZ` to overwrite the full file, then re-run the daily screen.

### `TelegramAlertError`

- **Cause:** Bot token invalid, chat ID wrong, network timeout, or Telegram API rate limit hit.
- **Fix:**
  1. Check `.env`: `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID` must both be set.
  2. Test the bot manually: `curl "https://api.telegram.org/bot<TOKEN>/getMe"`
  3. Verify chat ID: send a message to the bot, then `curl "https://api.telegram.org/bot<TOKEN>/getUpdates"`
  4. If rate-limited, wait 30s and re-run. The pipeline continues regardless — `TelegramAlertError` is caught in Step 11a and logged as `WARNING`.

### `BacktestDataError`

- **Cause:** Historical OHLCV data is missing for one or more symbols in the requested backtest date range (`start_date` → `end_date`). Most commonly triggered when `data/processed/{symbol}.parquet` does not cover the full date range.
- **Fix:** Run bootstrap for the affected symbols first: `python scripts/bootstrap.py --universe config`. Then verify coverage: `python -c "import pandas as pd; df=pd.read_parquet('data/processed/XYZ.parquet'); print(df.index.min(), df.index.max())"`. Shorten the backtest `--start` date if historical data simply doesn't exist.

### `"No module named X"` on fresh clone

- **Cause:** Project not installed in editable mode into the active venv.
- **Fix:**
  ```bash
  source .venv/bin/activate
  pip install -e ".[dev]"
  ```
  This registers all packages (`ingestion`, `features`, `rules`, etc.) so they are importable without path hacks. Required on every fresh clone or new venv.

## Performance Benchmarks

### Expected run times

| Operation | Symbols | Expected time |
|---|---|---|
| Daily run (incremental) | 500 | ~30 seconds |
| Daily run (incremental) | 2000 | ~2–3 minutes |
| Bootstrap (full history) | 500 | 5–15 minutes |
| Bootstrap (full history) | 2000 | 60–90 minutes |

### How to measure

```bash
# Feature-layer benchmark (runs feature pipeline on N symbols, prints per-symbol timing)
python scripts/benchmark_features.py

# Time a full daily run end-to-end
time python scripts/run_daily.py --date today

# Check last run duration from run_history
sqlite3 data/minervini.db \
  "SELECT run_date, duration_sec, symbols_screened FROM run_history ORDER BY run_id DESC LIMIT 5;"
```

### Warning signs — investigate if daily run exceeds 5 minutes

- **Benchmark `data/processed/` I/O:** Check if any symbol parquet has grown unexpectedly large (> 5000 rows). The `append_row()` strategy (read→concat→write) is O(n) in file size — symbols with 10+ years of daily data will slow down. Fix: use `parquet_store.write(..., overwrite=True)` to compact the file.
- **Benchmark network:** `yfinance` fetches are the most common bottleneck. If `--date today` is run before market close, yfinance may return stale data and retry. Always run after 15:35 IST.
- **Check for unexpected bootstraps:** `grep "bootstrapping" logs/minervini.log | wc -l`. If >10 symbols are bootstrapping on a normal daily run, feature files are being deleted or corrupted.
- **Check worker count:** `screener/pipeline.py` uses `ProcessPoolExecutor`. If the server is under heavy load (`htop`), reduce workers by passing `n_workers=4` or checking the config.
- **LLM latency:** If `llm.enabled: true`, Step 5b (narrative generation) adds ~1–3s per A+/A result. For 20+ A+ results, this can add 30–60s. Reduce `max_tokens` or set `only_for_quality: ["A+"]` in `settings.yaml`.
